# 同步逻辑说明

## 📊 改进后的同步逻辑

### Account 同步

#### 原来的逻辑
```
检查 CRM 中的 Books_Customer_ID 字段
    ↓
有值 → 更新 Books 记录
没值 → 创建新的 Books 记录
```

**问题**：如果 Books 中已经存在同名 Customer，会创建重复记录。

---

#### 改进后的逻辑 ✅
```
1. 在 Books 中查找是否存在同名 Customer
   （通过 company_name 查找）
    ↓
2. 找到了 → 更新该 Customer
    ↓
3. 没找到 → 创建新的 Customer
    ↓
4. 保存 Books ID 回 CRM
```

**优点**：
- ✅ 避免重复创建
- ✅ 自动关联已存在的记录
- ✅ 始终保持数据最新

---

### Product 同步

#### 原来的逻辑
```
检查 CRM 中的 Books_Item_ID 字段
    ↓
有值 → 更新 Books 记录
没值 → 创建新的 Books 记录
```

**问题**：如果 Books 中已经存在同名或同 SKU 的 Item，会创建重复记录。

---

#### 改进后的逻辑 ✅
```
1. 在 Books 中查找是否存在同名或同 SKU 的 Item
    ↓
   优先通过 SKU 查找（如果有 SKU）
    ↓
   如果 SKU 没找到，通过名称查找
    ↓
2. 找到了 → 更新该 Item
    ↓
3. 没找到 → 创建新的 Item
    ↓
4. 保存 Books ID 回 CRM
```

**优点**：
- ✅ 优先通过 SKU 匹配（更准确）
- ✅ SKU 没有时通过名称匹配
- ✅ 避免重复创建
- ✅ 自动关联已存在的记录
- ✅ 始终保持数据最新

---

## 🔍 查找逻辑详解

### Account 查找
```deluge
// 通过 company_name 查找
GET /contacts?organization_id={orgId}&company_name={Account_Name}
```

### Product 查找
```deluge
// 优先通过 SKU 查找
GET /items?organization_id={orgId}&sku={Product_Code}

// 如果没找到，通过名称查找
GET /items?organization_id={orgId}&name={Product_Name}
```

## 💡 为什么要改进？

### 场景 1：Books 中已有数据
- Books 中已经手动创建了一些 Customer/Item
- CRM 中这些记录没有 `Books_Customer_ID` / `Books_Item_ID`
- **旧逻辑**：会创建重复记录 ❌
- **新逻辑**：自动找到并更新已存在的记录 ✅

### 场景 2：数据迁移
- 从其他系统迁移数据到 Books
- CRM 中还没有对应的 Books ID
- **旧逻辑**：会创建重复记录 ❌
- **新逻辑**：自动关联并更新 ✅

### 场景 3：同步失败后重试
- 第一次同步成功，但保存 Books ID 到 CRM 失败
- CRM 中没有 Books ID，但 Books 中已有记录
- **旧逻辑**：会创建重复记录 ❌
- **新逻辑**：自动找到并更新 ✅

---

## ⚠️ 注意事项

### 1. 名称匹配规则
- **Account**：通过 `company_name` 精确匹配
- **Product**：优先通过 `sku` 匹配，其次通过 `name` 匹配

### 2. 多个匹配结果
如果找到多个匹配的记录，**只更新第一个**。

### 3. 更新策略
每次同步都会更新 Books 记录的所有字段，确保数据是最新的。

---

## 🎯 对比总结

| 方面 | 旧逻辑 | 新逻辑 |
|------|--------|--------|
| 依赖 | 依赖 CRM 中的 Books ID 字段 | 不依赖，自动查找 |
| 重复创建 | 可能创建重复记录 | 避免重复创建 ✅ |
| 数据迁移 | 需要手动关联 | 自动关联 ✅ |
| 容错性 | 差（保存失败会重复） | 好（自动恢复） ✅ |
| API 调用 | 1 次 | 2 次（查找 + 更新/创建） |
